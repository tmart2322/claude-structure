# Phase-builder — a reusable structure for shipping software in reviewed, resumable phases

This repo is a **portable scaffold** for running a software project as a sequence of **phases**, each one built by an orchestrated team of AI subagents and gated by an entry + exit review against your project's own principles. You copy it into a new repo, fill in a handful of files (your principles, your commands, your roadmap), and from then on `/phase-builder <n>` drives a whole phase end-to-end — planning it, building it in parallel, validating it, reviewing it against your tenets, and shipping it as one PR — while staying resumable across crashes and keeping the main agent's context lean.

It is **project-agnostic.** Nothing here knows or cares what you're building. The structure supplies the *process*; you supply the *project* by filling in `docs/`.

## How it works

### The thin-orchestrator model

`/phase-builder <n>` is a **thin orchestrator.** It does almost no work itself. Instead it dispatches **three background workflows**, each of which fans out to **specialized doer subagents** and returns a small, structured summary. The orchestrator reads only those summaries — never raw build/test logs — so a whole multi-day phase fits in one context window instead of blowing it out. (Doing everything inline in one agent is the exact failure mode this design exists to prevent.)

The three workflows:

| Workflow | When | What it does |
|---|---|---|
| **`phase-entry`** | Once, at the start | An `Explore` agent maps the repo and a `Plan` agent designs the build (chunks grouped into waves), in parallel. Then it writes and commits the **Stage-1 checkpoint**: the entry review, the task ledger, the exit-review skeleton, the operator-steps, and the backlog pull — all in one atomic commit. |
| **`phase-wave`** | Once per wave | Builds each chunk of the wave **in its own isolated git worktree, in parallel** (build → cover-with-tests → verify). Then a **single serial integrator** cherry-picks each verified chunk onto the phase branch, flipping its ledger line to `[x]` in the *same commit* as the code. Idempotent — re-run it with only the still-open chunks. |
| **`phase-exit`** | Once build work is done | Runs the automatable exit gates **in parallel**: the full CI-mirroring verify, live validation + dogfood, the tenet scorecard, a correctness review, a security review, and a docs reconcile. Returns the verdicts. |

### The doer subagents

The workflows dispatch ten role-based subagents (`.claude/agents/`). Each is pre-loaded with the *disciplines* of its role and reads `docs/PROJECT.md` for your project's specifics:

- **Builders** — `builder` (general/backend), `ui-builder` (the UI surface; delete if you have no UI).
- **Tester** — `tester` (real-boundary tests over mocks).
- **Reviewers** (read-only) — `reviewer-tenets` (scorecard + tenet violations), `reviewer-code` (correctness + simplification), `reviewer-security` (vulnerabilities).
- **Ops** — `verifier` (runs the gates, never fixes), `validator` (the keystone live end-to-end validation), `deployer` (ships to your target; delete if you don't deploy), `doc-syncer` (reconciles docs).

### The state machine

```
resume-check  →  ENTRY ⏸go/no-go  →  BUILD (wave by wave)  →  EXIT (gates G1–G11) ⏸acceptance ⏸PR ⏸merge
```

The `⏸` marks are the **operator seams** — the only four places a human is in the loop: approve the plan (go/no-go), do the final acceptance pass (G9), confirm the PR (G11), and merge it. Everything between is the agent's. A workflow can't reach these seams (workflows are non-interactive), which is exactly why they're the human's.

### Resumable by construction

Two committed files are the durable state, so a crash, a token-out, or a closed laptop loses nothing:

- **The task ledger** (`docs/phase-reviews/phase-<n>-tasks.md`) — one line per build chunk (`[ ]` / `[>]` / `[x]` / `[!]`), each with a `files:` glob. It's the source of truth for "what's left." The integrator flips a line to `[x]` *in the same commit* as that chunk's code, so the ledger and the tree never disagree.
- **The exit gate ledger** (in `phase-<n>-exit.md`) — the state of gates G1–G11. Resume = the first gate that isn't ✅/➖.

On every invocation, `/phase-builder` reconstructs phase state **from git**, not from the conversation: which chunks landed, which worktrees are orphaned, which gate to resume at. (See the *Resuming + crash recovery* section of the phase-builder skill.)

### The gate model + the backlog round-trip

Every phase is bracketed by two required reviews against `docs/TENETS.md` (see `docs/REVIEW.md`):

- **Entry review** — re-read the tenets, scope-check every deliverable against a north star, resolve blocking questions, pull the backlog, sharpen the definition of done. The operator gives go/no-go.
- **Exit review** — eleven gates (validation to terminal success, exit-criteria audit, tenet scorecard, design-tenet violations, dogfood proof, local-testability, docs synced, recurring-issues mechanized, operator acceptance, backlog reconciled, PR opened). Resumable and iterative.

Work surfaced but not finished never gets lost: the exit review proposes it as **backlog rows** (`docs/BACKLOG.md`), the operator approves them with a target phase, and the *next* phase's entry review **pulls** every row tagged for it into that phase's ledger. That round-trip is how deferred work survives a phase boundary instead of evaporating.

## Getting started

1. **Copy this structure into your repo** (the `.claude/` and `docs/` trees, and this README if you like). The target must be a git repo.
2. **Fill in the four core files** (see the next section): `docs/PROJECT.md`, `docs/TENETS.md`, `docs/PLAN.md`, and your first phase doc under `docs/phases/`.
3. **Restart your Claude Code session** so the new subagents in `.claude/agents/` load. (Skills and workflows hot-reload; **agents are read at session start**, so a freshly-added or renamed agent won't be dispatchable until you restart.)
4. **Run it:** `/phase-builder 1`. It cuts the `phase/1` branch, runs the entry review, and presents the plan for your go/no-go.

That's the whole loop. From there you approve the plan, let it build + validate, do your acceptance pass, and merge the PR.

## What you must fill in

Everything in angle brackets `<…>` in the docs is a placeholder. The four that matter, in order:

### 1. `docs/PROJECT.md` — the project profile *(most important)*

This is the single place the subagents learn the concrete facts about *your* repo. The system is generic; it reads the specifics from here. Fill in:

- **The command table** — typecheck, lint, test, the full gate sweep, the one-command local boot, the e2e command, build, deploy. *The verifier, validator, and gate-check skill all read these.* If a command doesn't exist yet, that's a Phase-1 deliverable — write it as `<TODO>` so it's visible.
- **Repo layout** — where code lives, so a builder doesn't rediscover it.
- **Hard invariants** — the rules a builder must never break (the checkable form of your design tenets).
- **Conventions** — branch naming, PR target, commit style, and whether the agent has autonomous deploy/sandbox authority.

### 2. `docs/TENETS.md` — your principles

Two layers: **north stars** (priorities, *ordered* — lower wins on conflict) and **design tenets** (invariants — never violated without an explicit conversation). The file ships with a curated, project-agnostic starter set; keep what fits, reword in your voice, cut what doesn't, and **add the handful unique to your project** (the file shows the shape of a good project-specific tenet). A few are marked **[load-bearing]** because the phase-builder process itself assumes them — keep those or an equivalent.

### 3. `docs/PLAN.md` — your roadmap

The doc index plus the **phase roadmap table** — one row per phase, sequenced so each phase's dependencies land first. Record your *ordering rationale*; future-you will want it.

### 4. `docs/phases/phase-<n>-*.md` — your phase docs

One per phase, from `docs/phases/_TEMPLATE.md`. See [Defining phases](#defining-phases). You don't have to write them all up front — write the next one or two; the backlog feeds the rest.

**Optional / as-you-go:** `docs/BACKLOG.md` (starts empty; seed it if you already have known work), `docs/ARCHITECTURE.md` (keep the *Risks* section current — the entry review reads it), and the mechanical-gate list in `PROJECT.md` (add gates as invariants emerge).

## Defining phases

A **phase** is a numbered, themed milestone — one coherent capability, shipped as one PR, bracketed by an entry and exit review. Each phase gets one doc (`docs/phases/phase-<n>-<theme>.md`) with five sections:

1. **Scope** — the *one themed capability* this phase delivers, its dependencies (which earlier phases must exist), and what's explicitly out of scope. If the scope reads like two unrelated capabilities, it's two phases.
2. **Deliverables** — the concrete things it builds, at the "what" level. Each should advance at least one north star (the entry review's scope check verifies this).
3. **Verification** — the scenarios that prove it works, written as **AI-run harnesses**, not operator to-do lists. These become exit gate G1. Each must be drivable end-to-end through the *real* surface to **terminal success**.
4. **Exit criteria** — the contract for "phase complete." Each bullet must be verifiably met at exit (gate G2).
5. **Tenets advanced** — which principles this phase moves forward, by name.

Two things to get right *while writing the phase doc*, because they're cheap to design in and expensive to bolt on:

- **Data ownership → schema + UI.** For every new entity or surface, ask **who owns it?** That one answer drives the schema (an owned entity carries its owner's FK) and the UI placement (a child entity lives *under* its parent, not as a global tab). The `phase-entry` Plan agent asks this too — but deciding it up front keeps the build on rails.
- **The operator's entry path.** For every capability, name the route *and the action* a human takes to start it from a fresh state. The "create / trigger / begin" affordance is **part of the deliverable** — a loop you can only enter by seeding a row or hitting an API directly is an incomplete phase, however complete its downstream steps.

A good way to derive exit criteria: take each Verification scenario and ask "what's the durable artifact or guarantee that proves this is *done*, not just *demoed*?" That's your exit criterion.

## Phase vs. wave vs. chunk

Three nested units. The distinction matters because **only the phase is a planning/review unit** — waves and chunks are execution detail.

| Unit | What it is | Who defines it | Gated? | Lifespan |
|---|---|---|---|---|
| **Phase** | One themed capability, shipped as one PR | You (a phase doc) | **Yes** — entry + exit review | ~1–2 weeks |
| **Wave** | A group of chunks that can build *in parallel* (their file ownership doesn't overlap) | The `phase-entry` Plan agent | No | One `phase-wave` run |
| **Chunk** | One unit of build work, owning a tight `files:` glob | The Plan agent | No (verified per-chunk) | One worktree |

- A **phase** is the milestone you reason about, review, and ship. It has a branch (`phase/<n>`), a doc, an entry review, an exit review with eleven gates, and exactly one PR.
- A **wave** is a *parallelism boundary inside a phase*. Chunks in the same wave run concurrently in isolated worktrees; **waves serialize where data forces it** — schema, migrations, and shared types land in an early wave (often "Wave 0 — serial") so later waves can build against them. There is **no review or gate per wave**; a wave is just "this batch can run at once." Wall-clock for a wave is the slowest single chunk, not the sum.
- A **chunk** is one build task with a non-overlapping `files:` glob (so two chunks never touch the same path — that's what makes parallel worktrees and crash recovery safe). Each chunk is built, tested, and verified independently, then integrated with an atomic code+ledger-flip commit.

In short: **you plan in phases; the system executes in waves of chunks.**

## How large should a phase be?

The right size is **one coherent, reviewable PR's worth of a single themed capability** — roughly **1–2 weeks** of work, typically **a handful to ~15 chunks across 2–4 waves**. Use these heuristics:

- **One theme.** If you can't name the phase's capability in a short phrase, or the name has an "and" joining two unrelated things, split it.
- **The PR test.** If the resulting PR would be too big for a human to review meaningfully, the phase is too big.
- **The ledger test.** If the task ledger has so many chunks you can't hold the shape in your head, split the phase. (Conversely, if a "phase" is two or three chunks with no real exit-review surface, it's probably just a feature — fold it into a neighboring phase.)
- **Dependencies point the knife.** The natural split is usually along a dependency seam: the substrate a later capability needs is its own earlier phase.
- **Front-load the foundation.** Early phases that everything builds on (the data layer, the local-boot command, the first CI gates, observability) earn being their own phase even if small — getting them wrong is expensive later. (See `docs/phases/phase-1-example.md`.)

When in doubt, **smaller and more frequent** beats large: each phase is a clean checkpoint with a green PR, so more phases means more recovery points and less blast radius per mistake.

## Running a phase

Once you've filled in the docs and restarted your session:

```
/phase-builder 1
```

What happens, and where you come in (the four `⏸` seams):

1. **Entry** — the workflow surveys + plans + commits the Stage-1 checkpoint, then shows you the entry review, the ledger (planned chunks vs. backlog-pulled), the risks, and the plan. **⏸ You give go/no-go.**
2. **Build** — wave by wave, it builds the open chunks in parallel worktrees and integrates the verified ones. Failed chunks come back as `needsAttention`; it re-dispatches them (a genuinely hard one runs on a stronger model). You watch the summaries; you don't read build logs.
3. **Exit** — it validates the whole loop **locally first**, then (if you deploy) on the real target to *terminal success*, then runs the parallel exit gates and writes the verdicts into the exit review. **⏸ You do the final acceptance pass (G9)** — "here's what I validated; what's not working?" Your notes become `(FB-n)` tasks the agent builds, then re-runs the touched gates.
4. **Ship** — once every gate is ✅/➖, it re-runs the full CI-mirroring sweep on the final tip, **⏸ confirms the PR with you**, opens it, and stages the closeout commit on the PR branch. **⏸ You merge.** The merge commit completes the phase.

Leave feedback any time by writing `> FEEDBACK: ...` inline in `phase-<n>-operator-steps.md`; the next `/phase-builder <n>` run triages it. The whole thing is resumable — if a session dies, just run `/phase-builder <n>` again and it recovers from git.

## Customizing the structure

- **Models.** Agent frontmatter uses tier aliases (`opus` / `sonnet` / `haiku`) so they don't go stale — tune per role (reviewers benefit from `opus`; the verifier is fine on `haiku`). The workflows escalate a `hard` chunk to `opus` and run the entry synthesis on `opus`.
- **Drop what you don't use.** No UI? Delete `ui-builder.md` (the wave workflow falls back to `builder` for every chunk). Don't deploy? Delete `deployer.md` and the deploy rows in `PROJECT.md`; the exit "deploy then validate on target" step collapses to "validate locally."
- **Your gates are yours.** The mechanical gates are defined in `docs/PROJECT.md` / your CI; the `gate-check` skill just runs them. Add a gate whenever an invariant gets violated once — a rule a human has to remember is a rule that slips.
- **The reviewers use built-in skills.** `reviewer-code` / `reviewer-security` invoke Claude Code's `code-review` / `security-review` / `simplify` skills if present, and review by hand otherwise.
- **Session restart after touching agents.** Adding or renaming anything in `.claude/agents/` requires a session restart before the workflows can dispatch it (agents are snapshotted at session start; skills and workflows hot-reload).

## The lessons baked in

The disciplines in these files aren't arbitrary — each one is a scar from a real failure. They transfer to any project; keep them even as you adapt everything else:

- **Read summaries, never raw output.** The orchestrator stays lean by trusting structured verdicts + committed ledger lines, not by re-reading the files the doers changed.
- **Terminal success, not side-effects.** A run isn't validated because a file got written or a route returned 200 — only when it reaches a terminal *success* state with the user-visible result actually present, driven through the real entry point on the real target. A mocked pass can't prove the real integration path.
- **The agent validates first; the human does final acceptance.** Validate locally first, exhaustively, to green; the operator is the last acceptance gate on an already-working build, never the one who discovers it's broken.
- **Atomic ledger-flip commits + full-range cherry-picks.** Code and its `[x]` land together so the tree and the ledger never disagree; the integrator cherry-picks a chunk's *whole* commit range (build + tests), never just the tip, so impl never lands without its tests.
- **Never `git reset --hard` with a dirty tree.** It silently eats uncommitted work; use `--soft`/`--mixed` + `git restore --staged`.
- **Re-sweep the full gate suite on the final tip before the PR.** The exit gates ran on an earlier commit; everything that landed after (feedback rounds, the default-branch merge, the closeout) is un-swept until you run the whole CI-mirroring sweep on the tip.
- **Nothing important lives only in prose.** Every surfaced-but-undone item becomes a backlog row with a target phase, or it gets lost.
- **Cite principles by name, not number.** Numbers are display order and break on reorder; names are stable.
